# 企业微信客服熔断与恢复说明

时间：2026-06-17

## 背景

线上 backend 曾出现 `Too many open files`，随后 `socket.accept() out of system resource`，导致 nginx upstream 超时。排查时发现旧客户端会高频轮询企业微信客服事件接口：

- `GET /api/wecom/proxy/kf/has-events`
- `POST /api/wecom/proxy/kf/sync_msg`

企业微信客服功能当前不是核心链路，为避免旧客户端未升级时继续触发企业微信上游调用、消息同步和后续 AI 回复处理，服务器已临时熔断该功能。

## 当前熔断行为

代码位置：

```text
/opt/lobster-server/backend/app/api/wecom_kf.py
```

默认状态：熔断关闭企业微信客服代理。

关闭后行为：

- 微信客服回调到达时，不再写入 `_kf_event_flags`
- `GET /api/wecom/proxy/kf/has-events` 永远返回无事件，并清理事件标记
- `POST /api/wecom/proxy/kf/sync_msg` 不访问企业微信上游，直接返回空消息列表
- `send_msg`、客服账号、客服人员、客户信息、二维码链接等代理接口返回 `503`

旧客户端仍然可以继续请求，但服务器不会再触发企业微信上游和后续处理链路。

## 恢复企业微信客服功能

在正式服务器执行：

```bash
sudo mkdir -p /opt/lobster-server/.runtime
sudo touch /opt/lobster-server/.runtime/wecom_kf_enabled
sudo systemctl restart lobster-backend
```

恢复后，`wecom_kf.py` 会检测到开关文件存在，企业微信客服代理重新启用。

## 再次熔断

如果恢复后发现 fd 增长、请求堆积、上游异常或 5xx 增多，执行：

```bash
sudo rm -f /opt/lobster-server/.runtime/wecom_kf_enabled
sudo systemctl restart lobster-backend
```

## 验证命令

### 1. 检查服务健康

```bash
curl -fsS https://bhzn.top/api/health
```

正常应返回：

```json
{"status":"ok","lan_ip":"10.1.0.14","mcp":{"reachable":true,"tools_count":14}}
```

### 2. 检查当前是否熔断

```bash
curl -sS https://bhzn.top/api/wecom/proxy/kf/has-events
```

熔断状态示例：

```json
{"has_events":false,"callback_path":"","ts":0,"disabled":true}
```

如果已恢复，返回里不应有 `disabled:true`。

### 3. 检查 backend 文件句柄

```bash
MAIN=$(systemctl show -p MainPID --value lobster-backend.service)
for p in $MAIN $(pgrep -P $MAIN); do
  [ -d "/proc/$p" ] && echo "pid=$p fd=$(ls /proc/$p/fd 2>/dev/null | wc -l)"
done
```

正常情况下 worker fd 应保持低位稳定。恢复企业微信后建议连续观察 10-30 分钟。

### 4. 检查近期是否还有 fd 耗尽

```bash
journalctl -u lobster-backend.service --since "30 minutes ago" --no-pager \
  | grep -Ei "Too many open files|socket.accept|out of system resource|Traceback|timeout|failed"
```

## 运行时开关说明

企业微信客服代理启用条件满足任意一个即可：

1. 环境变量：

```bash
LOBSTER_WECOM_KF_PROXY_ENABLED=1
```

2. 开关文件存在：

```text
/opt/lobster-server/.runtime/wecom_kf_enabled
```

当前推荐使用开关文件方式，恢复和熔断都不需要改代码，只需要重启 `lobster-backend`。

## 注意事项

- 这是服务器侧熔断，不要求客户端立刻升级。
- 恢复前建议确认客户端 OTA 降频已经覆盖一部分用户，否则旧客户端仍会高频轮询。
- 恢复后重点观察：
  - `lobster-backend` worker fd 是否持续上涨
  - nginx 是否出现大量 `499` / `504`
  - `journalctl` 是否再次出现 `Too many open files`
  - Postgres 连接是否异常堆积
- 如果恢复后异常重现，优先再次熔断，再分析企业微信链路和客户端轮询频率。
