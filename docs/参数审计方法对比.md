# 参数审计方法对比

## 我之前的检查方式（手动检查）

### 方法
1. **阅读代码**: 手动查看 `mcp/http_server.py` 中的 `_normalize_image_generate_payload` 和 `_normalize_video_generate_payload` 函数
2. **逐个模型检查**: 查看每个 `if` 分支，确认参数处理逻辑
3. **文档记录**: 创建文档记录每个模型的参数要求（如 `模型参数完整清单.md`）
4. **问题修复**: 根据发现的问题逐个修复

### 优点
- ✅ 可以深入理解代码逻辑
- ✅ 可以发现代码中的潜在问题
- ✅ 修复时能确保代码质量

### 缺点
- ❌ **不够系统化**: 依赖人工记忆，容易遗漏
- ❌ **无法覆盖所有模型**: 只检查了代码中明确处理的模型
- ❌ **无法验证枚举值**: 无法自动验证 `duration`、`aspect_ratio`、`resolution` 的枚举合法性
- ❌ **无法验证必填参数**: 无法自动检查必填参数是否被正确处理
- ❌ **效率低**: 需要逐个模型手动检查

### 实际检查内容
- ✅ 检查了代码中明确处理的模型（约 15-20 个）
- ✅ 修复了 `veo3.1` 的 `duration` 格式问题
- ✅ 添加了参数透传逻辑（保留用户传入的其他参数）
- ❌ 但无法保证所有 81 个模型都被正确检查

## xskill 的自动化审计方式（推荐）

### 方法
1. **获取模型列表**: 从 xskill API (`/api/v3/mcp/models`) 获取所有模型
2. **获取每个模型的 schema**: 调用 `/api/v3/models/{model_id}/docs` 获取 `params_schema`
3. **自动校验**: 
   - 检查必填参数是否在 normalize 结果中
   - 检查枚举参数的值是否合法（特别是 `duration`、`aspect_ratio`、`resolution`）
   - 检查参数类型是否匹配
   - 检查数值范围（minimum/maximum）
4. **生成问题清单**: 按风险等级（CRITICAL/HIGH/MEDIUM/LOW）排序
5. **按优先级修复**: 先修复高风险问题

### 优点
- ✅ **系统化**: 自动检查所有模型，不遗漏
- ✅ **准确性**: 基于真实的 API schema，不是猜测
- ✅ **可重复**: 可以定期运行，确保代码与 API 同步
- ✅ **优先级明确**: 按风险等级排序，优先修复关键问题
- ✅ **效率高**: 一次运行检查所有模型

### 实现
已创建脚本: `lobster-server/scripts/audit_model_params.py`

### 使用方法

```bash
# 运行审计（生成报告）
cd lobster-server
python scripts/audit_model_params.py

# 指定输出文件
python scripts/audit_model_params.py --output audit_report_20260315.md

# 使用自定义 xskill API 地址
python scripts/audit_model_params.py --xskill-url https://api.xskill.ai
```

### v2 改进（当前脚本行为）

- **仅审计 `category` 为 `image` / `video` 的模型**，跳过对话/音频等（无对应 normalize）。
- **按 schema 构造探测 payload**：`duration` 若在枚举中则选合法项（含字符串秒数、`4s`/`6s` 等）；`aspect_ratio`/`resolution` 有枚举则选 `16:9` 或首项；**必填项**用 `enum` / `examples` / `default` / 占位 URL 补齐。
- **校验只做一遍**：枚举与类型检查合并；**int 与字符串数字枚举**（如 `'5'` 与 `5`）视为匹配；**去重**相同 `(param, issue_type, 描述前缀)`。
- **并发拉取** `/api/v3/models/{id}/docs`，缩短总耗时。

报告示例：`docs/audit_report_latest.md`（随脚本运行更新）。

## 检查项对比

| 检查项 | 手动检查 | 自动化审计 |
|--------|---------|-----------|
| 必填参数检查 | ❌ 无法自动检查 | ✅ 自动检查 |
| 枚举值验证 | ❌ 手动查看文档 | ✅ 自动验证 |
| 类型检查 | ❌ 手动检查 | ✅ 自动检查 |
| 数值范围检查 | ❌ 无法检查 | ✅ 自动检查 |
| 覆盖所有模型 | ❌ 只检查代码中的 | ✅ 检查所有 81 个 |
| 可重复性 | ❌ 每次都要手动 | ✅ 一键运行 |
| 问题优先级 | ❌ 无法量化 | ✅ 按风险等级排序 |

## 建议的工作流程

### 1. 定期审计（推荐）
```bash
# 每周运行一次，确保代码与 API 同步
python scripts/audit_model_params.py --output audit_report_$(date +%Y%m%d).md
```

### 2. 修复前审计
在修复参数问题前，先运行审计脚本，了解所有问题：
```bash
python scripts/audit_model_params.py
# 查看报告，按 CRITICAL -> HIGH -> MEDIUM -> LOW 的顺序修复
```

### 3. 修复后验证
修复代码后，再次运行审计，确认问题已解决：
```bash
python scripts/audit_model_params.py
# 应该看到问题数量减少
```

## 总结

**我之前的检查方式**适合：
- 理解代码逻辑
- 修复已知问题
- 代码审查

**自动化审计方式**适合：
- 系统化检查所有模型
- 发现未知问题
- 确保代码与 API 同步
- 持续集成/持续验证

**最佳实践**：
- 使用自动化审计作为主要检查手段
- 手动检查作为补充（理解代码逻辑、修复复杂问题）
- 定期运行审计脚本，确保代码质量
